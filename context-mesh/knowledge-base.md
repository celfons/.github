## Security Remediation Patterns

### SQL Injection (SAST-SQLI)

- Always use parameterized queries or prepared statements.
- Never concatenate user input into SQL strings.
- Relevant rule: [`SECURITY/RULES/SAST-SQLI.md`](../SECURITY/RULES/SAST-SQLI.md)

### Cross-Site Scripting (SAST-XSS)

- Sanitize all user-supplied content before rendering as HTML.
- Use context-appropriate escaping: `html.escape()` in Python, `encodeURIComponent()` in JavaScript.
- Relevant rule: [`SECURITY/RULES/SAST-XSS.md`](../SECURITY/RULES/SAST-XSS.md)

### Remote Code Execution (SAST-RCE)

- Never pass user input directly to shell commands.
- In Python, use `subprocess` with an argument list and `shell=False`.
- Validate and whitelist all input before any execution call.
- Relevant rule: [`SECURITY/RULES/SAST-RCE.md`](../SECURITY/RULES/SAST-RCE.md)

## Minimal Fix Principle

AI agents must apply the **smallest possible change** that eliminates the vulnerability.

- Do not refactor, rename, or restructure unrelated code.
- Do not add tests unless the applicable rule explicitly requires them.
- Do not change behavior beyond what is required to fix the vulnerability.
- The fix must be surgical and auditable.

## Organizational Conventions

### Issue Label State Machine

Issues progress through these label states in order:

| Label | Meaning |
|---|---|
| `agent:ready` | Enriched and ready for agent execution |
| `triage-done` | Context files identified by triage agent |
| `plan-ready` | Action plan generated; awaiting human approval |
| `ai-approved` | Human approved plan; triggers executor job |
| `executing` | Executor agent is generating the solution |

### Workflow Data Flow

- **Triage → Planner**: Relevant `context-mesh/` file paths passed as a job output (JSON array).
- **Planner → Executor**: Plan stored in a GitHub issue comment delimited by
  `<!-- ai-agent-plan-start -->` / `<!-- ai-agent-plan-end -->` markers.
- **Executor → Repository**: Opens a PR on a branch named `ai-agent/issue-<number>-<timestamp>`.

### Pull Request Requirements

- All automated PRs must reference the originating issue with `Closes #<number>`.
- All automated PRs require human review before merging.
- Product code fixes and organizational rule updates must never be mixed in the same PR.
